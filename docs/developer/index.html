cms-ai — developer documentation (Sprint S41)

The cms-ai plugin adds an admin-only AI helper to the CMS post editor: an operator's prompt plus the page's own context (title / excerpt / content / CSS) is turned into a validated patch of CMS fields by the configured LLM (OpenAI- or Anthropic-protocol, chosen by model name), and — on demand — into a generated image written to the global CMS gallery. Text generation is stateless (the route returns a patch; nothing is persisted server-side); image generation persists the asset to the gallery immediately. The whole pipeline runs through LoopForge, a clean, import-clean engine that ships with no Flask, no SQLAlchemy, no session and no vbwd/plugin imports, so it can later be extracted as a pip dependency or git submodule.

Documents

  • LoopForge — the clean pipeline engine — what LoopForge is, the package layout, the StepTemplate / Flow / FlowRunner model, the LlmAdapter surface + select_adapter model-name routing + the two SDK implementations, the JSON validate/repair/retry loop, how to add a new step / LLM provider / image provider, and the import-clean oracle that enforces the boundary.
  • The prompt + field manifest — the admin-editable triple under ${VBWD_VAR_DIR}/assets/cms-ai/prompts/, how request_context and response_fields drive the request scope and the response JSON schema, how to add/remove a core field or an S77 custom-field key, how to ship a new template set / action, and the per-action requested-field sets.
  • Image generation — the ReplicateImageAdapter (Replicate / Black Forest FLUX black-forest-labs/flux-schnell), the config keys, the generate bytes → CmsImageService.upload_image → <img data-cms-image> → patch flow, the immediate-persist / save-on-Save timing, and the POST /generate-image route with its gating and token safety.

At a glance

LayerWhereResponsibility
Routecms-ai/routes.pyadmin-gated proxy; POST /generate, POST /generate-image
Servicecms-ai/services/cms_ai_generate_service.pybuild scope, derive schema, run flow, validate → patch
Servicecms-ai/services/cms_ai_image_service.pybytes → gallery upload → append <img> → patch
Asset loadercms-ai/services/prompt_asset_loader.pyresolve flow + template + manifest (var-dir override wins)
Engineloopforge/ (template, flow, adapter, image, …)pure, import-clean LLM/image pipeline

Two routes, both @require_admin + cms.manage, under the prefix /api/v1/plugins/cms-ai:

  • POST /api/v1/plugins/cms-ai/generate → {patch, provider, model}
  • POST /api/v1/plugins/cms-ai/generate-image → {patch, image}

The plugin declares a hard dependency on the cms plugin (it reuses CMS's global image gallery) and a soft, runtime-resolved dependency on S77 (custom-field defs), degrading to core fields when S77 is absent.
